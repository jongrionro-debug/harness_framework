# Architecture Decision Records

## 철학

이 프로젝트의 1차 MVP는 아래 원칙을 따른다.

1. 실제 운영 흐름을 빠르게 디지털화하는 것이 최우선이다.
2. 첫 구현은 단순해야 하지만, 나중에 다른 기관으로 확장할 수 있어야 한다.
3. 제품의 핵심 복잡도는 `다기관 데이터 분리`, `권한`, `세션 기반 기록`, `온보딩 생성 흐름`에 있다.
4. 따라서 화려한 분산 구조보다, 명확한 데이터 모델과 일관된 접근 통제가 더 중요하다.
5. 초기 성공 기준은 `배포 완료`가 아니라 `개발자 로컬 환경에서 핵심 흐름이 실제로 동작하는 것`이다.

---

### ADR-000: 1차 구현 성공 기준은 local-first MVP로 둔다

**결정**: 이 레포의 1차 구현 목표는 로컬에서 실제로 작동하는 MVP를 완성하는 것이다. 배포, 실메일 발송, 외부 운영 인프라 연결은 이후 단계로 미룬다.

**이유**:
- 현재 가장 중요한 것은 제품의 핵심 흐름이 문서대로 작동하는지 빠르게 검증하는 것이다.
- 인증, 기관 생성, 세션 관리, 강사 제출 흐름은 로컬에서도 충분히 검증 가능하다.
- 배포와 외부 운영 인프라를 초기에 같이 붙이면 구현 범위와 디버깅 복잡도가 불필요하게 커진다.

**트레이드오프**:
- 초기에 실제 운영 환경 검증은 늦어진다.
- 이메일 초대 같은 일부 플로우는 먼저 모의 또는 부분 연결 상태로 개발될 수 있다.

### ADR-001: Next.js App Router 기반 모놀리식 웹 앱 선택

**결정**: 1차 MVP는 `Next.js App Router + TypeScript + Tailwind CSS` 기반의 단일 웹 애플리케이션으로 구현한다.

**이유**:
- 기관 생성 온보딩, 운영자 웹, 강사 웹이 하나의 제품 안에서 자연스럽게 이어진다.
- 서버 렌더링, 인증 연동, 폼 처리, 파일 업로드, 관리 화면 구성이 한 프레임워크 안에서 해결된다.
- MVP 단계에서 프론트엔드/백엔드를 분리한 다중 저장소 구조보다 훨씬 빠르게 개발할 수 있다.

**트레이드오프**:
- 제품이 커지면 경계가 느슨해질 수 있다.
- 완전 분리형 프론트/백엔드보다 서비스 확장 시 유연성이 덜할 수 있다.

### ADR-002: Supabase를 Auth + DB + Storage의 기본 플랫폼으로 채택

**결정**: 인증, PostgreSQL, 파일 저장소는 `Supabase`를 기본 플랫폼으로 사용한다.

**이유**:
- 이 제품은 회원가입, 기관 생성, 이메일 초대, 첨부 문서 업로드가 초기에 모두 필요하다.
- MVP에서 Auth, DB, Storage를 각각 따로 조합하는 것보다 구현 속도가 빠르다.
- Postgres 기반이라 관계형 도메인 모델을 자연스럽게 설계할 수 있다.

**트레이드오프**:
- 특정 플랫폼 의존성이 생긴다.
- 나중에 다른 인프라로 옮길 때 마이그레이션 비용이 생긴다.

### ADR-003: 멀티기관은 단일 데이터베이스 + 공유 스키마 + organization_id 분리 방식으로 구현

**결정**: 기관별로 데이터베이스를 분리하지 않고, 하나의 데이터베이스에서 `organization_id`를 기준으로 데이터를 분리한다.

**이유**:
- 기관 생성 온보딩과 제품 내부에서의 다기관 확장 흐름에 가장 잘 맞는다.
- MVP 속도가 빠르다.
- 운영자, 강사, 세션, 기록 등 거의 모든 핵심 테이블에 자연스럽게 적용 가능하다.

**트레이드오프**:
- `organization_id` 누락은 매우 위험한 버그가 된다.
- 기관별 완전 물리 분리보다 보안 설계에 더 엄격해야 한다.

---

### ADR-004: 한 사용자는 1차 MVP에서 한 기관에만 소속된다

**결정**: 1차 MVP에서는 한 사용자가 여러 기관에 동시에 속하는 구조를 지원하지 않는다.

**이유**:
- 권한 모델이 단순해진다.
- 로그인 후 진입 흐름과 기관 생성 온보딩이 명확해진다.
- 초대, 역할 관리, 데이터 접근 범위를 쉽게 설명할 수 있다.

**트레이드오프**:
- 실제 현장에서 여러 기관에 걸쳐 일하는 사용자 요구를 바로 수용하지 못한다.
- 이후 다기관 소속을 열 경우 멤버십 모델과 UX를 다시 확장해야 한다.

---

### ADR-005: 기관 생성은 인증 후 최소 입력 온보딩으로 시작한다

**결정**: 기관 운영자는 회원가입 후 로그인하면 `기관 이름`과 `첫 번째 마을 이름`만 입력해 기관 운영 시스템의 초기 뼈대를 생성한다.

**이유**:
- 제품의 첫 경험을 가볍게 만든다.
- “입력하면 바로 내 운영 시스템이 생긴다”는 제품 인상을 줄 수 있다.
- 사업, 수업, 강사, 명단까지 한 번에 받지 않아 초기 이탈을 줄일 수 있다.

**트레이드오프**:
- 생성 직후 빈 운영 시스템 상태를 허용해야 한다.
- 생성 완료 후 다음 할 일 안내 UX가 매우 중요해진다.

---

### ADR-006: 기본 사업은 자동 생성하지 않는다

**결정**: 기관 생성 직후 사업은 자동 생성하지 않으며, 운영자가 직접 첫 사업을 만든다.

**이유**:
- 쓸모없는 기본 데이터가 쌓이지 않는다.
- 운영자가 자기 조직 언어에 맞는 사업 이름을 직접 정할 수 있다.
- 생성 완료 화면의 핵심 다음 액션을 `첫 사업 만들기`로 명확히 둘 수 있다.

**트레이드오프**:
- 생성 직후 대시보드보다 설정 안내 UX가 더 중요해진다.
- 완전 자동 생성형 제품보다 한 단계 더 수동 설정이 필요하다.

### ADR-007: 세션은 운영자가 미리 생성하고, 참여자 명단은 세션 시점 스냅샷으로 고정한다

**결정**: 수업 세션은 기관 운영자가 미리 생성하며, 세션 생성 시점의 참여자 명단을 복사해 세션 전용 스냅샷으로 보관한다.

**이유**:
- 출석부의 기준이 흔들리지 않는다.
- 과거 세션의 출석 대상이 이후 명단 수정 때문에 바뀌지 않는다.
- 강사와 운영자가 같은 세션 기준 데이터를 본다.

**트레이드오프**:
- 세션 생성 시 데이터 복사가 필요하다.
- 단순 참조형 구조보다 스키마가 조금 더 늘어난다.

---

### ADR-008: 제출은 최초 제출 시각과 최근 수정 시각을 분리한다

**결정**: `submitted_at`은 최초 제출 시각으로 유지하고, 제출 후 수정 시 `updated_at`만 갱신한다.

**이유**:
- “새로 제출된 기록”과 “나중에 수정된 기록”을 분리해서 해석할 수 있다.
- 완료율과 제출 건수 계산이 흔들리지 않는다.
- 운영자 대시보드에서 최근 제출 기록과 최근 수정 기록을 각각 보여주기 쉽다.

**트레이드오프**:
- 시각 필드의 의미를 UI에서 분명히 보여줘야 한다.
- 단순한 단일 `updated_at` 모델보다 설명해야 할 개념이 하나 더 생긴다.

---

### ADR-009: 강사 초대는 이메일 초대 링크 방식으로 처리한다

**결정**: 기관 운영자는 강사를 이메일 초대 링크 방식으로 초대한다.

**이유**:
- 운영자가 비밀번호를 직접 만들고 전달하는 방식보다 안전하다.
- 초대 수락 기반 흐름이 제품 확장에 유리하다.
- 역할 부여와 기관 소속 연결을 명시적으로 처리할 수 있다.

**트레이드오프**:
- 초대 링크 만료, 재발송 정책이 필요하다.
- 이메일 전달 실패 같은 운영 이슈를 다뤄야 한다.

---

### ADR-010: 운영자 대시보드는 최근 제출 기록과 최근 수정 기록을 분리한다

**결정**: 대시보드에는 `최근 제출 기록`과 `최근 수정 기록`을 각각 별도 영역으로 둔다.

**이유**:
- 새로 들어온 기록과 나중에 변경된 기록은 운영 의미가 다르다.
- 제출 중심 관리와 수정 추적을 동시에 만족할 수 있다.
- 최근 수정 기록에 강사 수정과 운영자 수정을 함께 포함해 운영 변경 흐름을 더 잘 볼 수 있다.

**트레이드오프**:
- 대시보드 정보량이 약간 늘어난다.
- 수정 이벤트를 식별할 수 있는 메타데이터 관리가 필요하다.

---

### ADR-011: 세션은 강사와 참여자 없이 먼저 생성할 수 있다

**결정**: 운영자는 강사와 참여자 명단이 준비되지 않아도 세션을 먼저 생성할 수 있다. 강사는 세션별 관리 화면에서 나중에 할당하고, 참여자는 생성 후에도 세션별로 추가할 수 있다.

**이유**:
- 실제 운영에서는 일정과 수업이 먼저 확정되고 강사/명단이 나중에 확정되는 경우가 있다.
- 세션 생성이 강사 초대 수락이나 초기 명단 준비에 막히면 운영자의 첫 경험이 무거워진다.
- 세션별 대시보드를 두면 운영자가 생성 후 부족한 정보를 보완할 수 있다.

**규칙**:
- `sessions.teacher_id`는 nullable이다.
- 참여자 0명 세션 생성은 허용하지만, 강사 제출은 참여자 snapshot이 1명 이상일 때만 허용한다.
- 제출 후 참여자를 추가하면 `submitted_at`은 유지하고 `updated_at`만 갱신한다.
- 제출 후 추가된 참여자는 출석 기록이 생기기 전까지 UI에서 `미입력`으로 표시한다.
